# GW-FOC-Lab 双人任务管理与激励文档

> 适用对象：我和同门两个人
> 项目方向：FOC 电机控制实验平台
> 核心原则：低压力、可持续、有记录、有反馈、有成就感
> 版本：v0.1

---

## 1. 为什么要做这个文档

我们两个人做这个 FOC 项目，不是导师安排的硬任务，也不是短期必须完成的课题，而是为了未来找工作、积累工程经验、做一点自己真正感兴趣的东西。

但是我们平时工作日时间少，经常被导师、组会、实验、杂活挤压，行动力也不一定稳定。如果只靠热情，很容易出现三种情况：一开始聊得很兴奋，后面没人动；遇到困难卡住之后就放一边；做了一些东西但没有记录，最后简历和展示都说不清楚。

所以这个文档的目的不是给自己增加压力，而是建立一个很轻量的管理机制，让项目能持续往前走。我们的目标不是每天都做很多，而是每周至少留下一个可见的小成果。

---

## 2. 项目总目标

本项目拟建设一个面向学习和工程实践的 FOC 电机控制实验平台，内容包括嵌入式端的 FOC 控制算法、Linux/Python 上位机数据采集与可视化、实验数据记录与分析，以及后续物联网和智能状态识别扩展。

项目最终希望形成一个可以在 GitHub 展示的完整工程，包括代码、实验记录、运行截图、演示视频、原理说明和阶段总结。该项目既服务于我们对控制、嵌入式、Linux、物联网和 AI 的兴趣，也服务于未来求职时的项目经历展示。

---

## 3. 项目定位

本项目不是竞赛项目，不要求短时间内做出很复杂的系统；也不是纯理论学习项目，不能只停留在看资料和写笔记。

本项目的定位是：

**一个能逐步迭代的 FOC 电机控制实验平台。**

第一阶段只追求最小闭环：

**电机能稳定转起来 → 能采集数据 → 能显示数据 → 能调参数 → 能保存实验结果。**

后续再逐步扩展 Linux、Web、物联网、音频检测、深度学习等方向。

## 4. 双人分工

### 4.1 控制与嵌入式方向负责人

主要负责 MCU 端、电机控制和底层算法相关工作。

典型任务包括：

* 电机驱动板调试
* 编码器读取
* SVPWM / FOC 基础实现
* 电流环、速度环、位置环控制
* PID 参数整定
* 电机保护逻辑
* 串口/CAN/USB 数据输出
* 嵌入式端代码整理与注释

### 4.2 Linux / 上位机 / 数据方向负责人

主要负责上位机、数据采集、可视化、文档和后续智能化方向。

典型任务包括：

* Python 串口通信程序
* 实时曲线显示
* CSV 数据保存
* 实验数据分析脚本
* Linux 部署
* Web 控制面板
* MQTT / IoT 扩展
* 音频采集与状态识别预研
* GitHub 仓库维护与 README 编写

### 4.3 共同任务

以下任务两个人共同参与：

* 每周确定一个小目标
* 每周更新一次项目进展
* 重大问题一起讨论方案
* 每完成一个阶段，一起录制或整理演示材料
* 每月做一次阶段复盘

## 5. 项目阶段路线

### v0.1：项目初始化

目标：把仓库、文档和基本计划搭起来。

完成标准：

* GitHub 仓库建立
* README 初版完成
* docs 文件夹建立
* 本任务管理文档加入仓库
* 明确第一块硬件和第一阶段目标

### v0.2：电机开环运行

目标：先让电机在基础驱动下转起来。

完成标准：

* 确认电源、驱动板、电机连接正确
* 电机可以开环转动
* 记录硬件连接方式
* 拍摄一段运行视频
* 记录遇到的问题和解决方法

### v0.3：编码器与速度采集

目标：能够读取电机角度和速度。

完成标准：

* 编码器数据可以正常读取
* 能计算速度
* 能通过串口输出速度数据
* Python 可以接收并打印数据
* 保存一份测试 CSV

### v0.4：速度闭环控制

目标：实现基础速度闭环。

完成标准：

* 可以设置目标速度
* 电机能够跟随目标速度
* 能观察实际速度和目标速度差异
* 记录 PID 参数
* 画出一次阶跃响应曲线

### v0.5：Python 上位机可视化

目标：实现最小上位机。

完成标准：

* 实时显示目标速度和实际速度
* 实时显示基础状态量
* 可以保存实验数据
* 可以通过上位机修改目标速度
* README 中加入运行截图

### v0.6：实验分析与文档整理

目标：让项目开始具备展示价值。

完成标准：

* 至少完成 3 组速度阶跃实验
* 输出响应曲线
* 分析超调、稳态误差、响应时间
* docs 中补充实验记录
* 整理一版项目阶段总结

### v0.7：Linux / IoT 扩展

目标：把项目扩展到 Linux 或边缘端。

完成标准：

* 程序可以在 Ubuntu / 树莓派 / Linux 环境运行
* 支持长期数据采集
* 支持后台运行或脚本启动
* 初步支持 MQTT 或 Web 展示

### v0.8：智能分析预研

目标：探索 AI、音频或故障识别方向。

完成标准：

* 明确一个智能分析问题
* 采集一批数据
* 完成一个最小分类或异常检测 demo
* 形成一份预研笔记

## 6. 每周任务机制

我们不采用高压计划，而采用“每周一个小成果”的机制。

每周只要求完成一个核心任务。任务必须足够小，小到即使工作日很忙，周末抽一点时间也能推进。

### 每周任务格式

每周任务需要写成下面这种形式：

```text
本周目标：
负责人：
预计耗时：
完成标准：
输出物：
当前状态：未开始 / 进行中 / 已完成 / 暂停
```

示例：

```text
本周目标：Python 串口接收 MCU 输出的速度数据
负责人：A
预计耗时：2 小时
完成标准：能够接收一行包含 target_speed 和 actual_speed 的数据
输出物：serial_reader.py、测试截图、README 说明
当前状态：未开始
```

## 7. 任务粒度要求

为了防止任务太大导致拖延，每个任务必须满足以下条件：

1. 单个任务预计不超过 3 小时。
2. 任务必须有明确输出物。
3. 任务不能只写“学习 FOC”这种模糊目标。
4. 如果任务超过 3 小时，就必须拆分。
5. 每个任务完成后，必须在仓库里留下痕迹。

不合格任务示例：

```text
学习 FOC
研究电机控制
做一个上位机
搞一下 Linux
看看深度学习
```

合格任务示例：

```text
整理 FOC 坐标变换公式到 docs/foc_theory.md
写一个 Python 串口接收 demo
画出第一次速度阶跃响应曲线
把实验数据保存为 CSV
补充 README 的硬件连接说明
```

## 9. 激励机制

我们不搞复杂的 KPI，但可以搞一个轻量积分机制，用来给自己正反馈。

### 9.1 积分规则

| 行为          | 分数 |
| ----------- | -: |
| 完成一个小任务     | +2 |
| 提交一次有效代码    | +1 |
| 写一份有效文档     | +1 |
| 上传一张实验截图或曲线 | +1 |
| 解决一个卡点问题    | +2 |
| 做一次阶段总结     | +3 |
| 录制一次演示视频    | +3 |
| 连续两周无任何推进   | -2 |
| 任务完成但没有记录   |  0 |

### 9.2 积分用途

积分不是为了比较谁强谁弱，而是为了给自己反馈。

建议每月统计一次。两个人总积分达到一定值，可以给自己一个小奖励。

| 月度总积分  | 奖励建议            |
| ------ | --------------- |
| 10 分   | 吃一顿好的           |
| 20 分   | 买一个项目小模块        |
| 30 分   | 半天不干正事，专门玩项目    |
| 40 分以上 | 整理一次 GitHub 展示版 |

## 10. 防摆烂规则

我们承认会忙、会累、会拖延，所以需要设置最低推进要求。

### 最低要求

每周至少满足下面任意一项：

* 提交一段代码
* 写一段文档
* 整理一张图
* 跑一次实验
* 记录一个问题
* 看一篇资料并写 5 行总结

也就是说，即使这周很忙，也不能完全没有痕迹。

### 忙碌周规则

如果本周被导师 push、组会、实验或其他事情压满，可以启动“忙碌周模式”。

忙碌周只要求完成一个 15 分钟任务，例如：

* 整理一个 TODO
* 提交一次 README 修改
* 记录一个 bug
* 把实验照片上传到 docs
* 写下下次要做什么

忙碌周不算失败，只要留下痕迹就算推进。

## 11. GitHub 使用规则

### 11.1 提交要求

每次提交尽量写清楚做了什么，不要只写 `update`。

推荐格式：

```text
feat: add serial speed logger
fix: correct encoder speed calculation
docs: update hardware wiring guide
exp: add speed step response data
refactor: clean protocol parser
```

### 11.2 分支规则

前期可以简单使用 main 分支，不必一开始搞复杂流程。

当代码逐渐复杂后，可以增加：

```text
main        稳定版本
dev         日常开发版本
feature/*   单个功能开发
```

### 11.3 仓库中必须保留的内容

项目不是只放代码，必须保留以下内容：

* 代码
* README
* 硬件连接说明
* 实验数据
* 实验曲线
* 问题记录
* 阶段总结
* 演示视频链接或截图

## 15. 第一阶段建议任务

第一阶段不要贪多，只做项目初始化和最小可运行链路。

### 第 1 周

目标：建立仓库和基本文档。

完成标准：

## 16. 我们的约定

1. 不因为一周没做就放弃项目。
2. 不追求一开始做得很高级，先追求能跑通。
3. 不做只有口头讨论、没有记录的任务。
4. 不把任务写得太大，所有任务都要能拆成小块。
5. 每周至少留下一个痕迹。
6. 遇到卡点先记录，再讨论，不要闷着不动。
7. 项目服务于我们自己，不是为了给自己制造新的压力。
